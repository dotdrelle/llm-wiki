## Scope

This release reduces provider-bound latency without changing the knowledge
model: shared provider throttling, bounded in-job concurrency, token-budgeted
build batches, BM25-first build context, and per-run instrumentation.

## Exit criteria

- Shared RPM budget: `tests/rate-limiter.test.ts` starts two independent Node
  processes against one workspace and verifies their combined request starts
  respect the same rolling provider window.
- Coarse-grain parallelism: the manager runtime scheduler runs independent plan
  tasks concurrently, exposes `taskId` on events for the Run/Task graph, enforces
  write locks, handles cancellation, patching and child failure, and replays
  deterministic workflow projections (`llm-wiki-manager/src/runtime/runner.test.js`,
  `llm-wiki-manager/src/runtime/store.test.js`).
- Targeted production jobs: `production_start_job` accepts scoped `inputs`,
  `templates`, and `deliverables`; targeted build jobs lock their deliverables
  so non-conflicting jobs can run in parallel while conflicting writes wait
  (`agent-wiki-production/test_production_mcp_server.py`).
- Fine-grain concurrency: build batches and ingest section calls respect
  `limits.maxInFlightRequests`, with the shared RPM bucket still bounding the
  global provider start rate.
- Token-budgeted batches: `planSlotBatches` uses
  `limits.targetInputTokensPerCall`; `build.slotBatchSize` is only a compatibility
  ceiling. A template with six slots plans as one batch when the token budget
  allows it.
- Retrieval: build context uses BM25 by default through
  `retrieval.buildStrategy: bm25`; chat and MCP search keep rerank behavior.
- Instrumentation: each traced command emits `trace:summary` and prints LLM,
  embedding, rerank, throttle, provider-latency and wall-time totals at command
  end.

## Measurement note

The instrumentation is now the source of truth for real provider gains. For a
representative workspace, capture the summary printed by `wiki build`, `wiki
ingest`, or `wiki export` and compare:

- wall time before/after enabling `limits.maxInFlightRequests`;
- provider throttle time before/after the shared RPM bucket;
- rerank calls before/after BM25-first build context;
- build batch count before/after token-budgeted planning.

The release should not claim a production speedup number until a real provider
run has written those summaries in `.wiki/logs/*`.
